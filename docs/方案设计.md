# 方案设计

## 一、问题背景

用户在观看外语演讲、国际会议、技术分享、网课和直播时，经常需要快速理解连续语音内容。传统翻译工具通常依赖用户复制文本或上传文件，无法直接覆盖真实桌面观看场景，也难以处理实时音频中的识别误差、上下文缺失和术语不一致问题。

本项目面向 Windows 桌面端，开发一款 AI 同声传译助手。系统直接捕获电脑正在播放的单向音频流，通过 AI 语音识别、上下文翻译和自动纠错能力，将外语内容转换为中文字幕，并以悬浮窗形式展示。

## 二、设计目标

### 1. 流畅度

- UI 不因语音识别或翻译任务阻塞。
- 用户启动后可以在短时间内看到临时字幕反馈。
- 临时字幕、正式字幕和修正字幕状态清晰，不造成明显跳屏。
- 悬浮窗支持置顶、拖动、透明度和字号调节，适配视频、会议和网课场景。

### 2. 准确度

- 通过滑动窗口和 VAD 分段提升语音识别稳定性。
- 翻译时结合最近几句上下文，避免逐句机械翻译。
- 对技术、会议和课程场景维护术语表，保证专业词汇一致。
- 对最近若干条字幕执行回看修正，降低实时翻译的上下文误差。

### 3. 现代 UI

- 主控制窗口采用现代工具型界面，突出状态、音频源、模型配置和历史记录。
- 悬浮字幕窗口保持简洁，避免干扰用户观看内容。
- 字幕状态通过颜色、透明度和短暂高亮区分。
- 设计优先服务高频使用体验，不做营销式首页。

## 三、整体架构

```text
Windows 系统音频
  -> 音频采集层
  -> 音频处理层
  -> 语音识别层
  -> 翻译层
  -> 字幕修正层
  -> 字幕状态管理
  -> 桌面 UI 层
```

### 音频采集层

基于 Windows WASAPI Loopback 捕获当前默认播放设备输出的声音。用户无需打开麦克风，也无需把视频导入本软件，只要目标内容正在系统中播放，应用即可获取音频流。

当前 MVP 使用 `soundcard` 封装系统音频采集能力，支持枚举 loopback 设备、录制短音频保存为 wav，并支持持续输出固定时长的音频块，用于后续 VAD 和 ASR 消费。

### 音频处理层

音频处理模块负责完成音频切片、音频环形缓冲、Silero VAD 分段和滑动窗口缓存。MVP 阶段采用 ONNX Runtime + Silero VAD 路线，不引入 PyTorch / torchaudio。系统持续缓存最近数秒音频，并通过 `speech_start` / `speech_end` 事件为后续 ASR 提供语音段边界。

### 语音识别层

语音识别模块通过适配器封装，当前支持 mock ASR 和百度智能云短语音识别标准版 HTTP 接口。mock 模式用于无 API Key 时验证桌面演示和自动化测试；真实 ASR 模式将 VAD 输出的语音段转换为 16k 单声道 PCM，调用百度短语音识别接口，并返回统一的 `AsrResult`。系统会根据源语言自动选择百度 `dev_pid`，英文演讲默认使用英语模型，中文输入默认使用普通话模型。后续如果需要本地部署，可继续新增 SenseVoice / FunASR ONNX 或其他本地模型适配器，而不影响上层字幕链路。

### 翻译层

翻译模块不会只翻译单句，而是将当前字幕段、最近上下文和术语表一起发送给翻译模型，生成更自然的中文表达。对于尚未稳定的内容，先生成临时字幕；语音段稳定后生成正式字幕。

### 字幕修正层

系统保留最近若干条字幕记录。当后续上下文提供更明确的语义时，修正模块判断是否需要回写前文字幕。修正时根据 `segment_id` 定位原字幕段，并产生 `segment.update` 事件。

### 桌面 UI 层

桌面 UI 包含主控制窗口和悬浮字幕窗口。主控制窗口用于开始、暂停、配置音频源、选择模式和查看历史记录；悬浮字幕窗口用于覆盖在视频或会议窗口上方，实时展示中文或双语字幕。

## 四、字幕事件协议

为了支持实时展示和历史回修，系统内部使用事件驱动的字幕协议。

### 临时字幕

```json
{
  "type": "segment.partial",
  "segment_id": "seg_001",
  "source_text": "The model uses attention to",
  "zh_text": "这个模型使用注意力……"
}
```

### 正式字幕

```json
{
  "type": "segment.final",
  "segment_id": "seg_001",
  "source_text": "The model uses attention to align tokens.",
  "zh_text": "该模型使用注意力机制来对齐 token。"
}
```

### 修正字幕

```json
{
  "type": "segment.update",
  "segment_id": "seg_001",
  "old_zh_text": "该模型使用注意力机制来对齐 token。",
  "zh_text": "该模型通过注意力机制对齐各个 token。",
  "reason": "terminology_consistency"
}
```

### 状态管理规则

系统为字幕段维护独立状态容器，而不是只向界面追加文本。状态容器按 `segment_id` 保存字幕段，并遵循以下规则：

- `segment.partial`：新增或更新临时字幕，用于低延迟展示。
- `segment.final`：确认字幕段，表示当前语音片段已经稳定。
- `segment.update`：回写已有字幕段，记录旧文本、修正原因和新文本。
- 已经确认或修正的字幕不会被迟到的临时事件覆盖。
- 未命中已有字幕段的修正事件会被视为异常，避免修正变成误追加。

当前 MVP 已加入模拟字幕流。模拟流会按时间间隔产生 `partial`、`final` 和 `update` 事件，驱动主窗口与悬浮字幕窗口同步更新，用于在接入系统音频、ASR 和翻译前验证用户体验。

## 五、MVP 范围

第一阶段优先验证完整链路，而不是一次性追求所有 AI 能力：

1. 初始化项目结构和中文文档。
2. 实现主控制窗口和悬浮字幕窗口。
3. 实现系统音频捕获，并可保存测试音频。
4. 定义字幕事件协议和状态管理。
5. 用模拟字幕流驱动 UI，验证实时展示和历史修正效果。
6. 接入 ASR 适配器，支持 mock 与真实 AI 语音识别。
7. 接入中文翻译。
8. 加入上下文修正和术语一致性。
9. 完善打包、演示和 README。

## 六、演示方案

最终 demo 计划使用一段英文技术分享或公开课作为输入。演示时打开视频或会议音频，启动助手后展示：

- 悬浮窗实时显示中文字幕。
- 临时字幕快速出现。
- 正式字幕在语音段稳定后确认。
- 某条历史字幕根据后续上下文被自动修正并高亮。
- 主窗口展示双语历史记录和修正记录。
